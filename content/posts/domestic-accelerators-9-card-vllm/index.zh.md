> **TL;DR** — 同一 Qwen2.5-7B、同一负载（`in=512 / out=128`），每卡一张，扫并发到饱和。峰值 output 吞吐从高到低：**昆仑芯 P800 3455 · 阿里 PPU 3407 · 沐曦 C500 3222 · 昇腾 910B4 1679 · RTX 4090D 1602 · 海光 K100-AI 1483 · 天数 MR-V100 428 · V100 427 · 寒武纪 MLU370 18**（tok/s）。两张 98 GB 的国产卡在高并发下把消费级 4090D 甩开 **2 倍**——大显存装得下大 KV Cache，这正是吞吐型服务的胜负手。但更锋利的发现是：**沐曦只用一个 32 GB 切片（满算力）就冲上领奖台**，这意味着对 7B dense 模型，瓶颈在算力/带宽，**而不是容量**。

这是 [NVIDIA 基线那篇](../index.md)（英文）的续集——同一套方法学，现在铺到七张国产加速卡外加两张 NVIDIA 卡做参照。

## 为什么要做这个横评

公开的 LLM 推理跑分几乎都是单卡孤立跑、用作者手头恰好有的硬件。跨厂商、**同一模型 / 同一负载 / 同一测量工具**的数据本来就稀缺——而对中国国产加速卡（昆仑芯、昇腾、海光、沐曦、寒武纪、天数、阿里 PPU），更是近乎空白，因为你得真机接触到一整机架全都插着这些卡的集群。我恰好有。所以我把九张卡并排跑，每个数字都附上确切命令。真机数字造不了假，这就是价值所在。

## 测试口径

| 维度 | 值 |
|---|---|
| 模型 | `Qwen2.5-7B-Instruct`，dtype **float16**，每张卡上都是逐字节一致的 checkpoint（config / index / tokenizer 的 sha256 + safetensors 分片大小全部匹配） |
| 负载 | `random` 数据集，固定 **input=512 / output=128** token，`--ignore-eos`，`--seed 42` |
| 扫并发 | max-concurrency ∈ {1, 4, 16, 64, 128, 256}，吞吐不再涨（<3%）即停 |
| 工具 | 每张卡都用 vLLM 自带的 `vllm bench serve` |
| 集群 | 67（全部国产 + NVIDIA）、183（昇腾 910B4） |

负载刻意跟 NVIDIA 那篇完全一致，好让数字能叠起来对比。大卡在并发 64 时还没饱和，所以这一轮扫到 256——不扫到每张卡各自的天花板，就排不出"谁每 token 更划算"。

## 方法学

- 每次扫描前先跑一轮 warmup（并发 4、20 条 prompt）并丢弃。
- `--ignore-eos` 强制每条请求恰好吐 128 个 output token → TPOT 干净可比。
- num-prompts 随并发放大（50 / 100 / 200 / 400 / 512 / 768）。
- TTFT = 首 token 时刻 − 发送时刻；TPOT = 除首 token 外每个 output token 的时间；吞吐 = output token 数 / 墙钟时间。
- **模型一致性是验证过的，不是假设的。** 每张卡都服务同一个 Qwen2.5-7B release——靠哈希 `config.json` / `model.safetensors.index.json` / `tokenizer.json` 并比对 safetensors 分片大小来确认。"模型名字一样"不等于"字节一样"，我校了字节。

> **诚实口径，先摆在前面：** 模型统一了，但**引擎版本没统一**——每家厂商都发自己的 vLLM fork（版本见下表）。V100 根本跑不了现代引擎（见基线那篇的"软件悬崖"发现），国产卡也只能跑自己那套 build。所以这是*"每张卡在它各自的实际最佳状态下"*，而非*"同一引擎、不同硅片"*。我把这一点点破，而不是埋掉。真正的跨厂商横评，要么诚实，要么就是营销。

## 数字说明什么 〔观点待核 — David 用自己的话改写〕

**1. 大显存是吞吐武器——但要在高并发下。** 两张 98 GB 卡（昆仑芯 P800、阿里 PPU）峰值 3400+，是 24 GB 4090D 的两倍。不是因为它们硅片快 2 倍——并发 1 时它们只快约 1.4–1.9 倍。差距随并发拉开，因为 98 GB 装得下巨大的 KV Cache，让调度器把几百条请求塞进同一个 batch。吞吐型服务，先是显存的游戏，然后才是算力的游戏。

**2. ……但容量不是故事的全部。** 沐曦 C500 用一个 **32 GB 切片**（满算力、只用卡的一半显存）拿到季军，把 4090D 甩开 2 倍。7B 模型在 fp16 下约 14 GB 权重；32 GB 在中等 batch 下留给 KV 还绰绰有余。所以容量只在你真需要一个巨型 KV 时才兑现——极高并发，或大模型。对 7B dense，算力 + 带宽才是约束项，而沐曦有。

**3. 昇腾 910B4 ≈ 4090D。** 1679 vs 1602——910B4 略胜消费级 4090D。一张扎实、不花哨的推理卡。

**4. "对标 V100"名副其实。** 天数 MR-V100 峰值 428 tok/s；一张真 NVIDIA V100 是 427。以对手命名，也精确命中了对手。

**5. 小卡跑 7B dense 是个陷阱。** 寒武纪 MLU370-X4（23 GB）峰值仅 18 tok/s，并发 16 时 TTFT 爆到 85 *秒*。这卡是为小模型 / embedding 造的，不是 7B dense 服务。对的工具，错的活。

## 32B 测试：只有大卡才启动得了

98 GB 最有力的论据不是 7B 的数字——而是一个 62 GB bf16 的 **Qwen3-32B 能单卡装下**。4090D（24 GB）、V100（32 GB）、昇腾 910B4（32 GB）都装不下单卡，得跨多卡张量并行。两张 98 GB 卡直接就装下了。

![Qwen3-32B——昆仑芯 P800 vs 阿里 PPU](compare-32b.png)

| 卡 | 显存 | Qwen3-32B 峰值 tok/s | vs 自己的 7B |
|---|---|---:|---|
| 昆仑芯 P800 | 98 GB | **601** | 3455 → 601（慢 5.7×） |
| 阿里 PPU | 98 GB | 383 | 3407 → 383（慢 8.9×） |

〔观点待核〕7B 上两家难分伯仲（3455 / 3407）；32B 上 P800 明显领先（601 vs 383）。更大的模型更吃算力和显存带宽，P800 在这两头都更有余量。单卡 32B 能力本身就是国产大卡的价值主张——没有 TP 拓扑，没有互联税。

## 自己复现

```bash
# 起服务（在各家 GPU pod 内；模型走本地/NFS 路径，绕开下载噩梦）
vllm serve /models/Qwen2.5-7B-Instruct --served-model-name qwen2.5-7b \
  --dtype float16 --gpu-memory-utilization 0.90 --max-model-len 4096 --port 8000

# 扫一个并发档（对 1,4,16,64,128,256 逐个重复）：
vllm bench serve --backend vllm --model qwen2.5-7b \
  --tokenizer /models/Qwen2.5-7B-Instruct --host 127.0.0.1 --port 8000 \
  --dataset-name random --random-input-len 512 --random-output-len 128 --ignore-eos \
  --percentile-metrics ttft,tpot,itl --metric-percentiles 90,99 --seed 42 \
  --max-concurrency 64 --num-prompts 400 --save-result --result-filename c64.json
```

各家的 quirk、pod YAML、完整数据都在仓库里。

## 让我意外的地方 / 踩坑（AI 编不出来的部分）

- **一张"空闲"的沐曦卡却调度不上。** index-0 的 C500 显示 0% 利用率、无进程——是真空的——可就是没有 pod 能排到它上面。根因是**驱动层**配额泄漏：`mx-smi` 报 `total_vram_quota=3813` 而不是满值 `64549` MiB，因为前一个 GPU 虚拟化（sGPU）租户异常退出、没归还显存配额。卡*看起来*空闲，但驱动认为它已满，于是调度器正确地拒绝往上放东西。**可迁移的教训：在国产 sGPU 栈上，"空闲"不等于"可用"——你得看驱动的显存*配额*，不是看利用率数字。** 回收泄漏的配额是平台/运维层的操作，不是租户该在共享集群上随手做的——所以作为使用方，正确动作是*识别*出这个差异、把负载引到健康的卡上。这种"利用率会骗人、要看配额"的 bug，是 NVIDIA 生态很少甩给你的一类。
- **卡住我的是跨子网带宽，不是 GPU。** 把 14 GB 权重弄到远端节点（172.28.4.x）——从模型源（10.100.121.x）过去——不管走 NFS、SSH 隧道还是 pod 网络，都只有 ~1–10 MB/s；三个 pod 并发读同一个 NFS checkpoint 更是把它拖到近乎停摆。解法平淡但彻底：每张卡都从它节点本地已有的 checkpoint 起服务。
- **调度器层本身就是一类"为什么跑不起来"。** 这些集群在厂商 device plugin 前面跑了一个开源的异构 GPU 调度器；在它被指到你的负载有权使用的资源池之前，pod 就一直 `Pending`——知道了就是小事，不知道能困你一小时。在异构集群上,"这张卡是空的"和"调度器会把它给你"是两个独立的问题。

## 局限 & 接下来测什么

- 各厂商引擎版本不同（无法避免——每张卡只跑自己的 fork）。模型统一且逐字节校验过；引擎是"每张卡各自的最佳"。
- 单卡、单模型、单一 input/output 形状。还没做跨厂商 TP、长上下文、量化。
- **perf-per-¥ 〔David 填：国产卡价格我没有可靠来源，故意留白〕** — 真正决定平台采购的排名是每元 token 数，不是 token/s。这些国产卡我没有可信的市场价，所以我不编。你有真实报价，就填这里。
